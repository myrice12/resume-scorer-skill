# 评分规则

无论是调用本地脚本，还是手动进行判断，都统一使用这套评分规则。

## 简历 `resume`

- `completeness`：个人简介、经历、项目、技能、教育等关键模块是否完整且易于扫描。
- `impact`：bullet 是否体现了动作、范围和量化结果，而不只是职责罗列。
- `clarity`：结构是否简洁、时间顺序是否清晰、是否适合快速浏览。
- `targeting`：简历是否明确贴合目标岗位，或与目标 JD 的关键词保持一致。
- `credibility`：时间、工具、职责边界和业务规模是否足够具体、可信。

## 项目经历 `project`

- `context`：读者能否快速理解问题背景、目标用户或业务目标。
- `ownership`：是否清楚区分出候选人本人主导或亲自完成的部分。
- `technical_depth`：是否体现架构选择、技术约束或非平凡实现细节。
- `impact`：是否说明了覆盖范围、性能提升、业务收益或上线规模。
- `clarity`：是否遵循紧凑的情境-行动-结果叙述方式。

## 岗位描述 `jd`

- `role_clarity`：岗位名称、级别、团队信息和职责目标是否清楚。
- `specificity`：任职要求是否具体、可筛选，而不是泛泛而谈。
- `structure`：岗位职责、硬性要求和加分项是否清楚分层。
- `attractiveness`：JD 是否说明了候选人为什么值得投递这个岗位。
- `screening_efficiency`：候选人能否在较短时间内完成自我判断，而不用读一大段清单。

## 修改建议优先级

优先做那些改动最小、但信号提升最大的修改：

1. 先补结构缺失。
2. 再补量化结果。
3. 然后对齐目标岗位词汇。
4. 证据层足够扎实之后，再去润色措辞。
