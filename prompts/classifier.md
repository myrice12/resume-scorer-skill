# 输入分类器

## 任务

识别用户提供的文本属于哪一类输入，并提取后续评分需要的基础信号。

---

## 支持的输入类型

| 类型 | 说明 |
|------|------|
| `resume` | 完整简历或简历主体内容 |
| `project` | 单段项目经历、项目介绍、项目 bullet |
| `jd` | 岗位说明、岗位职责、任职要求 |
| `resume+jd` | 简历和目标 JD 一起提供 |
| `project+jd` | 项目经历和目标 JD 一起提供 |

---

## 识别规则

### 识别为 `resume`

满足以下特征中的多个：

- 包含“工作经历 / 实习经历 / 项目经历 / 教育背景 / 技能 / 个人简介”
- 包含邮箱、电话、LinkedIn、GitHub 等联系信息
- 出现多段按时间排序的经历
- 同时出现职位名称、公司名、学校名、技能列表

### 识别为 `project`

满足以下特征中的多个：

- 文本集中描述一个项目或一个模块
- 常见结构是“背景/目标 → 做了什么 → 结果”
- 内容长度通常短于完整简历
- 强调个人负责内容、技术方案、项目结果

### 识别为 `jd`

满足以下特征中的多个：

- 出现“岗位职责 / 职责 / 任职要求 / 要求 / 加分项 / 我们希望你”
- 语气是招聘方视角，而不是候选人自述
- 包含 title、level、requirements、nice to have、福利等内容

### 识别为 `resume+jd` 或 `project+jd`

- 用户明确说“和这个 JD 对比”
- 同时提供候选人文本和岗位文本
- 文本中存在明显的“候选人视角 + 招聘方视角”两种内容

---

## 需要提取的关键信号

无论是什么类型，都尽量提取：

- 语言：`中文 / English / mixed`
- 目标岗位：如“后端工程师 / 数据分析师 / 产品经理”
- 关键词：技能、工具、领域词、岗位词
- 量化结果：百分比、用户量、金额、耗时、覆盖范围
- 明显缺失项：如无结果、无时间、无技能、无职责边界

如果有 JD，对候选人文本额外提取：

- 已匹配关键词
- 缺失关键词
- 最值得补齐的岗位词汇

---

## 输出格式

按以下结构输出分类结果：

```text
类型：{resume / project / jd / resume+jd / project+jd}
语言：{中文 / English / mixed}
目标岗位：{如果能判断}
核心关键词：{3-10 个}
量化结果：{列举}
主要缺失：{列举}
后续评分规则：{resume / project / jd}
是否需要匹配度评分：{是 / 否}
```
