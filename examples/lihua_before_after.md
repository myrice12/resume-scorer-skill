# 李华示例：优化前后对比

> 示例展示 `resume-scorer` 如何评估项目经历，并给出可直接落地的修改建议。

---

## 原始项目经历

```markdown
负责公司内部审批平台的开发，使用 React 和 FastAPI，支持多个团队使用。
```

### 问题分析

- 背景不清楚：不知道这个平台解决什么问题
- 主导度不清楚：只说“负责开发”，看不出具体职责边界
- 技术深度不足：没有说明关键设计或约束
- 结果不清楚：只说“支持多个团队”，没有量化价值

---

## 评分示例

```text
总评：这段项目经历有基础信息，但还不足以支撑高质量简历中的项目亮点。

总分：58/100

分项评分：
- 背景上下文：52/100
- 个人主导度：60/100
- 技术深度：55/100
- 影响力：48/100
- 清晰度：75/100

主要问题：
- 没说明平台解决的业务问题
- 没说明你本人负责了哪些关键模块
- 缺少上线范围或效率提升数据

优先修改建议：
1. 补一句背景，说明这是为了解决什么审批流程问题
2. 明确你负责的是权限模型、表单迁移还是流程引擎
3. 增加结果数据，例如覆盖团队数、流程时长下降比例、人工处理时间减少情况
```

---

## 优化后项目经历

```markdown
主导内部审批平台核心模块开发，使用 React 与 FastAPI 重构跨团队审批流程，负责权限模型设计、历史表单迁移和 MVP 上线。
平台上线后覆盖 12 个业务团队，将人工审批处理时间缩短 35%。 
```

### 优化点

- 第一条补了项目背景和个人贡献边界
- 第二条补了上线范围和量化结果
- 技术栈保留，但不再只是“报菜名”

---

## 改写边界

如果原始文本里没有真实数据，不应直接编造。正确做法是写成：

```markdown
平台上线后覆盖 [待补充：团队数] 个业务团队，将人工审批处理时间缩短 [待补充：百分比]。
```
